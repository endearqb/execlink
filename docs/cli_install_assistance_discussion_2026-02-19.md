# AI CLI 安装协助讨论记录（2026-02-19）

## 1. 讨论背景

用户希望确认：`ExecLink` 是否可以在应用内协助安装 Kimi、Claude Code、Gemini CLI、Codex CLI，并给出可落地方案。

## 2. 结论

可以实现。  
推荐采用“可确认的半自动安装”方案，而不是静默全自动安装。

## 3. 已确认的安装相关事实

- `Kimi Code CLI`：官方安装脚本通常不要求用户先手动准备 Python 环境（脚本会处理 uv + 安装流程）。
- `Claude Code`：官方推荐原生安装器/winget，不强依赖用户先手动安装 Node.js。
- `Gemini CLI`：使用 npm 安装时需要 Node.js（官方要求 Node.js 20+）。
- `Codex CLI`：使用 npm 安装时需要 Node.js；官方提示 Windows 仍属 experimental，建议优先 WSL 场景。

参考来源见：`docs/cli_install_methods_2026-02-19.md`

## 4. 产品方案建议（用于后续实现）

### 4.1 安装前检查

- 检查 `Node.js` / `npm` / `pwsh` / `winget` / `WSL`。
- 根据目标 CLI 给出依赖差异提示：
  - `Gemini`、`Codex`：提示依赖 Node.js。
  - `Codex`：Windows 下提示 WSL 建议。

### 4.2 安装入口

- 每个 CLI 提供：
  - `复制安装命令`
  - `打开官方说明`
  - `一键安装（可选）`

### 4.3 一键安装交互规范

- 执行前弹窗，明确：
  - 将执行的命令
  - 命令来源域名/发行方
  - 可能触发的权限或网络行为
- 用户确认后再执行。
- 安装过程应可见（便于登录与错误定位）。

### 4.4 安装后回写与复检

- 安装完成后自动执行 `where <cli>` 复检。
- 自动刷新 UI 检测状态。
- 失败时给出错误详情、日志路径、官方文档链接。

## 5. 安全边界（必须）

- 仅允许白名单命令模板（按 CLI 固定）。
- 仅允许白名单域名（官方站点）。
- 对远程脚本安装（如 `irm|iex` / `curl|bash`）必须二次确认。
- 禁止自由输入任意命令执行。

## 6. 推荐实施阶段

1. 先上线“安装前检查 + 双入口（复制命令/打开文档）+ 复检”。
2. 再上线“一键安装（半自动确认执行）”。
3. 最后视反馈补企业环境策略（代理/离线包/执行策略说明）。
